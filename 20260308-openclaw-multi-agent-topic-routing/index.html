OpenClaw 多 Agent Topic 路由配置踩坑

问题描述 / 背景介绍

在 Telegram 群组中部署 OpenClaw，需要实现不同 Topic 的消息路由到不同的 agent 处理。

环境信息：

  • OpenClaw Gateway
  • Telegram Group (Forum 模式)
  • 多 agent 配置：blog_agent, ops_agent, chat_agent

问题现象：

  • 默认配置下，所有 topic 的消息都路由到同一个 agent
  • 无法实现按 topic 隔离的专业化处理
  • 用户在 Blog topic 提问，却触发 chat agent 回复

(此处应有截图：Telegram 群组 topic 列表)

解决方案概述

通过配置 bindings 规则，将 Telegram topic_id 与特定 agent 绑定，实现精准路由。

核心思路：

  1. 识别每个 topic 的唯一 ID
  2. 在 channel 配置中添加 bindings 映射
  3. 重启 Gateway 使配置生效
  4. 验证各 topic 消息路由到正确的 agent

Step 4 - 测试多 Agent 隔离

在不同 topic 发送测试消息：

  1. Blog topic (ID: 2) 发送 “写一篇测试文章”

    • 预期：blog_agent 响应，生成写作卡片

  2. Ops topic (ID: 3) 发送 “检查服务器状态”

    • 预期：ops_agent 响应，执行健康检查

  3. General topic (ID: 1) 发送普通消息

    • 预期：默认 agent 响应，或无响应（如未配置）

Troubleshooting / Notes

Note: 如路由不生效，检查以下可能原因：

  1. topic_id 不准确 - Telegram 的 topic_id 与显示顺序可能不一致
  2. JSON 语法错误 - 确认 bindings 配置格式正确
  3. Gateway 未完全重启 - 等待 10-30 秒让配置完全加载
  4. Bot 权限不足 - 确保 Bot 有读取 topic 元数据的权限

Note: 如后续新增 topic，需同步更新 bindings 配置并重启 Gateway。

总结

  • 使用 bindings 配置可实现精准的 topic → agent 路由
  • topic_id 必须准确，建议在测试环境先验证
  • 多 agent 架构可大幅提升专业化处理能力
  • 配置变更后务必重启 Gateway 并测试验证
